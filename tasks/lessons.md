# Lessons

- Verifier l'etat reel du depot avec `flutter analyze` et `flutter test` avant d'appliquer un lot suppose, car le hotspot annonce peut deja etre resolu localement.
- Quand on travaille en petits lots BMad, toujours relier le lot local a l'etat global du projet: dire clairement ce qui est nettoye, ce qui reste bloquant pour "application fonctionne", et la prochaine etape de roadmap.
- Quand un modele impose des invariants temporels (`updatedAt >= createdAt`), les fixtures de test doivent mettre a jour les deux champs ensemble pour eviter les flakes entre run cible et run global.
- Quand l'utilisateur veut revenir au flux BMAD formel, ne pas router vers `Quick Dev` par defaut: verifier d'abord que `_bmad-output/planning-artifacts` contient un vrai `*epic*.md`, puis suivre `Sprint Planning -> Sprint Status -> Create Story -> Validate Story -> Dev Story -> Code Review`.
- Quand le workflow BMAD configure la communication en francais ou que l'utilisateur le rappelle explicitement, repondre en francais sans melanger avec de l'anglais hors contenu technique cite.
- Quand l'utilisateur pense avoir ouvert tous les acces via une commande locale comme `codex --full-auto`, expliciter que les permissions des tools restent distinctes et continuer le flux sans supposer un bypass implicite du sandbox.
- Quand l'utilisateur demande de preparer la story suivante d'un lane explicite (par exemple "apres 1.11"), ne pas laisser `create-story` auto-selectionner la premiere story `backlog` globale si cela saute vers un autre epic; prioriser la continuite du lane demande ou expliciter le changement d'epic avant d'agir.
- Quand un identifiant de story peut etre ambigu (`1.5` vs `1.15`), relire le contexte BMAD recent et la derniere story manipulee avant de lancer une revue ou d'annoncer un verdict sur la mauvaise cible.
- Quand l'utilisateur precise qu'il n'a pas de credentials applicatifs valides, ne pas presenter le flow signe-in comme acquis: traiter les fichiers de test comme de simples indices, documenter le blocage, et separer clairement ce manque d'acces d'une regression produit.
- Quand j'utilise l'outil navigateur/remote-debug du poste, ne pas le presenter comme une dependance Playwright du projet sans verifier d'abord les fichiers du repo (`package.json`, config e2e, etc.).
- Quand l'utilisateur demande explicitement de continuer "avec les agents", verifier d'abord si la session expose un vrai outil de delegation; sinon annoncer ce manque clairement, basculer en analyse parallele locale, et continuer sans promettre un sous-agent indisponible.
- Quand une revue BMAD ne remonte qu'un ecart de bookkeeping directement corrigeable dans les artefacts (`File List`, statut, tracker), appliquer la correction et resynchroniser la story au lieu de s'arreter a un simple constat.
- Quand l'utilisateur veut ouvrir un nouvel epic mais demande d'abord une phase de brainstorming, suspendre `create-epics-and-stories`, lancer la decouverte avant la redaction, et ne pas fabriquer un Epic suivant sans intention produit explicitee.
- Quand le workflow BMAD est conduit en francais, ne pas laisser les titres de story, user stories ou acceptance criteria repartir en anglais par habitude; tout le contenu de travail soumis au user doit rester en francais sauf terme technique necessaire.
- Quand l'utilisateur rappelle "Parle francais", maintenir tout l'echange en francais sans exception, y compris les validations, transitions de workflow et resumes intermediaires.
- Quand l'utilisateur injecte explicitement un skill BMAD avec instruction de charger le workflow complet, lire integralement le `workflow.md` vise avant toute execution et aligner la revue sur ses etapes exactes.
- Quand un `correct-course` BMAD est approuve et que l'utilisateur demande de "faire ce qui est recommande", executer directement la sequence recommandee sur les artefacts approuves au lieu de reposer une question de selection deja resolue, sauf blocage reel non deduisible des documents.
- Quand un lot remplace des textes hardcodes par `AppLocalizations`, realigner dans le meme changement les harnais de test sur un `MaterialApp` localise et des assertions basees sur les cles l10n reelles, sinon on fabrique de faux echecs de revue.
- Quand `sprint-status.yaml` et les story records montrent un epic deja termine mais que les artefacts canoniques parlent encore de cet epic au futur, lancer d'abord `correct-course` pour resynchroniser `prd.md`, `architecture.md`, `ux-guidance-minimale.md` et `epics.md` avant de reecrire le lane suivant ou de relancer un readiness check cible.
- Quand une story depend d'une cible publique deja deployee, verifier le bundle reel servi (`index.html`, `manifest.json`, `main.dart.js`) et pas seulement le repo local, les captures precedentes ou la doc; un drift de variable GitHub Pages peut laisser une URL publique "vivante" mais branchee sur un host placeholder.
- Pour une build Flutter web qui charge `flutter_dotenv`, ne pas inferer la config runtime depuis une simple recherche de chaine dans `main.dart.js`: les constantes de fallback du code source y restent compilees. Verifier `assets/.env` et, si besoin, `auth/v1/settings` pour confirmer la vraie cible publique servie.
- Quand l'acceptance criterion demande seulement de prouver l'acces d'un utilisateur externe explicitement invite, ne pas surcontraindre la recette en exigeant un compte dedie ou des credentials repo-owned; un compte perso peut suffire s'il correspond bien au pilote invite et si la preuve est documentee sans exposer les secrets.
- Pour un flux Supabase `signUp` avec confirmation email, ne pas conditionner le feedback UI a `response.user`; l'absence de `session` suffit a indiquer une confirmation en attente. Et transmettre explicitement `SUPABASE_AUTH_REDIRECT_URL` via `emailRedirectTo`, sinon on peut envoyer un email valide mais rediriger vers le mauvais `Site URL`.
- Quand un correctif est seulement present dans le workspace local, ne pas conclure qu'un redeploy GitHub l'a teste: GitHub Actions ne voit que les commits pousses. Re-verifier `git status` sur les fichiers du lot avant d'interpreter un retour utilisateur provenant d'une cible publique.
- Quand un build Flutter casse dans des fichiers `app_localizations_*.dart`, verifier d'abord les fichiers source `.arb` et leurs placeholders avant de patcher le Dart genere: le build regenere ces fichiers et ecrase sinon le faux correctif.
- Avant d'annoncer qu'une URL publique reflete bien la derniere implementation, comparer explicitement l'etat du code officiel (`main` / remote) avec le worktree local: un poste local sale peut contenir le bon comportement alors que l'instance publiee sert encore un commit plus ancien.
- Quand une QA de shell Flutter est re-ecrite, la relancer sur viewport desktop et mobile: les tests alignes avec l'etat officiel revelem souvent de vrais overflows UI que les anciens tests obsoletes masquaient.
- Quand on veut versionner quelques fichiers dans un dossier ignore par Git, des exceptions sur les fichiers seuls ne suffisent pas toujours: il faut aussi reautoriser le dossier parent (ex. `!/web/icons/`) puis verifier avec `git ls-files` que les artefacts attendus sont bien suivis avant de compter sur CI/CD.
