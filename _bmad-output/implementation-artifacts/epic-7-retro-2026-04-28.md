## Ce qui a bien marché

1. **Rapidité d'exécution.** 10 stories denses (bugs critiques + RGPD + i18n + tests) menées à terme. C'est le signe que l'équipe a gagné en maturité de process depuis l'Épic 6.

2. **Pattern 'Diagnostiquer avant de coder'.** Stories 7.1, 7.2 et 7.3 ont démarré par une analyse statique des causes racines. En 7.2, les 3 bugs d'Elo (double calcul, pas de persistance Supabase, UI non rafraîchie) ont été identifiés et corrigés en une seule story sans rework.

3. **Code review systématique `[Patch]`/`[Defer]`.** Chaque story a eu une review formelle avec classification explicite. Bugs évités avant deploy : memoization Provider (7.0), `context.mounted` manquant après await (7.4), double-tap "Accepter" non protégé (7.7), fichier de test absent du repo alors que marqué 'créé' (7.2).

4. **Helper partagé `test/helpers/localized_widget.dart`.** Créé en 7.3, réutilisé sans modification en 7.4, 7.5, 7.6, 7.7. Pattern DRY appliqué naturellement.

5. **Harness Supabase centralisé (7.9).** Code d'initialisation dupliqué dans deux fichiers de test extrait en classe partagée. Problème de compte de test inexistant (`test_1776892399910_958`) détecté et corrigé au passage.

6. **Gestion de la dette différée via `deferred-work.md`.** Chaque review alimente le fichier avec contexte (raison du report, impact). La story 7.0 a soldé la dette de l'Épic 6 en premier — pattern à maintenir.

## Ce qui n'a pas bien marché

1. **67 tests en échec ignorés sur toute l'Épic 7.** Chaque story note 'failures pré-existantes inchangées' sans investigation. On ne sait pas si c'est du code cassé ou des tests invalides. DataMigrationService est un module potentiellement sensible (données utilisateur).

2. **Plusieurs choses n'ont pas abouti au premier coup.** AC2 story 7.3 (détecter quit/refresh pendant import) non implémenté. Compte de test inexistant en 7.2. 5 strings i18n manquées en première passe de 7.6. Test d'intégration marqué 'créé' mais absent du repo (7.2, détecté en review).

3. **Incohérence sprint-status / fichier story.** 7.1 avait `Status: review` dans son fichier alors que sprint-status indiquait `done`. Même pattern qu'en Épic 6 avec la story 6.2. Corrigé post-rétro.

4. **Deux action items Épic 6 non adressés.** `architecture.md` (déviation router guard 6.4) et procédure vérification mobile avec rate limit actif n'ont pas été traités pendant l'Épic 7.

5. **Credentials de test dans l'historique git.** `test_credentials.txt` était tracké par git. Corrigé en 7.9 via `.gitignore` + `git rm --cached`, mais le mot de passe reste dans l'historique — rotation du compte de test requise.

6. **Déploiement non confirmé à la clôture.** Il n'est pas établi que les features Épic 7 (notamment la ConsentGatePage RGPD) sont actives pour les utilisateurs pilotes.

## Leçons clés

**Leçon 1 — Diagnostiquer avant de coder réduit le rework.**
Ce pattern appliqué en 7.1, 7.2, 7.3 a évité plusieurs aller-retours. À systématiser dès le début de chaque story impliquant un bug ou une régression.

**Leçon 2 — La code review `[Patch]`/`[Defer]` a une valeur concrète.**
Sur 10 stories, les reviews ont détecté des bugs réels avant deploy (memoization, mounted guards, fichiers manquants). Ce processus doit être maintenu et formalisé dans le template de story.

**Leçon 3 — 'Pré-existant hors scope' n'est pas une absolution.**
67 tests en échec persistent sans propriétaire. Une dette sans propriétaire ne disparaît pas. Il faut assigner une session dédiée de diagnostic.

**Leçon 4 — Les credentials de test dans git sont un risque structurel.**
`git rm --cached` ne suffit pas — le mot de passe reste dans l'historique. Mettre en place un `.env` avec fallback sur `test_credentials.txt` (fait en 7.9) mais aussi roter le compte de test.

**Leçon 5 — La synchronisation sprint-status / fichier story doit être disciplinée.**
Un accord d'équipe simple suffit : `Status: done` dans le fichier `.md` avant de mettre `done` dans sprint-status.
