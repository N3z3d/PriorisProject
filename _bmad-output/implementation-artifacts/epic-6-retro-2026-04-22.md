## Ce qui a bien marché

1. **`WebAuthCallbackStabilizer` — fondation extensible.** Le commit `4f5120e` a posé une architecture solide que 6.4 a pu étendre sans refactoring majeur. Les méthodes exposées via `@visibleForTesting` ont permis 20 tests directs sans mock complexe.

2. **Sprint Change Proposal — processus maîtrisé.** Quand le bug `#sb → Route non trouvée` a été découvert le 2026-04-20, l'épic a été rouvert proprement avec une story bornée, sans dérive de scope.

3. **Analyse en trois causes racines distinctes.** La story 6.4 a séparé : fragment `#sb` non reconnu (cause 1), timing Flutter web / Dart (cause 2), PKCE verifier absent sur navigateur in-app mobile (cause 3). Cela a permis des fixes ciblés et testables.

4. **Couverture de tests auth rigoureuse.** Cas nominal, `#sb`, `#sb-...`, régression `#access_token=...`, cas négatifs — la matrice de test sur code auth critique était complète.

5. **`AppConfig` comme source de vérité unique** pour l'identité pilote. Pas de duplication de métadonnées entre `LoginHeader`, `HomePage`, `web/index.html`, `web/manifest.json`.

## Ce qui n'a pas bien marché

1. **Épic rouvert — `#sb` non anticipé.** Le fragment Supabase sans `=` ni `&` n'était pas couvert par les critères d'acceptance de 6.1 à 6.3. Découvert uniquement via preuve publique réelle. Les tests unitaires et l'intégration repo-owned ne peuvent pas attraper ce type de cas.

2. **Rate limit Supabase récurrent.** Le blocage `429` sur envoi d'emails est apparu le 2026-04-19 et a re-bloqué le 2026-04-21. Aucune procédure alternative documentée. La preuve mobile est restée partielle.

3. **Deux items de dette technique différés.** La duplication de logique (`_isSupabaseCallbackRoute` / stabilizer) et le couplage cross-layer (static infra consommé par la présentation en contournant Riverpod) ont été identifiés mais non résolus — hors scope d'une story bug-fix.

4. **Incohérences documentaires.** Story 6.2 toujours à `ready-for-dev` dans son fichier alors que le sprint-status dit `done`. Aucun artefact de rétro Epic 5 présent malgré une référence dans 6.1. Dossier `planning-artifacts` vide. Déviation architecture 6.4 (router guard) non reportée dans `architecture.md`.

5. **Pas de test fonctionnel utilisateur avant clôture.** Le test manuel de Thibaut post-épic a révélé plusieurs régressions et bugs critiques invisibles aux tests unitaires : schema mismatch Supabase (`habits.category`), Elo non mis à jour, import massif sans feedback.

## Leçons clés

**Leçon 1 — Matrice de callback auth obligatoire dès la création de story.**
Tout AC impliquant un flow auth web public doit spécifier les variantes : PKCE / implicite / fragment route-like / navigateur in-app mobile. La preuve repo-owned ne suffit pas — il faut une recette sur device réel avec email réel, datée, avant clôture.

**Leçon 2 — Validation fonctionnelle sur compte non-créateur.**
La clôture d'un Epic doit inclure un test utilisateur sur un compte normal (pas le compte créateur). Les bugs de schema Supabase et de logique core ne sont visibles qu'en condition réelle. Les mocks ne voient pas les dérives de schéma.

**Leçon 3 — Les items de dette différés ont une durée de vie courte.**
La dette technique différée doit être planifiée en début de l'épic suivant, pas laissée ouverte indéfiniment. Elle bloque les refactors et les tests.

## Découvertes significatives post-épic (test Thibaut 2026-04-22)

Ces découvertes ont été faites après la clôture officielle de l'Épic 6 et définissent directement le périmètre de l'Épic 7.

### Bug critique 1 — Schema mismatch habitudes
```
PostgrestException(message: Could not find the 'category' column of 'habits'
code: PGRST204)
```
La fonctionnalité habitudes est cassée en production. Désynchronisation entre le modèle Dart et le schéma Supabase réel.

### Bug critique 2 — Elo / classement non fonctionnel
Les scores Elo ne se mettent pas à jour après les comparaisons. Les lots ne changent pas. La fonctionnalité core du produit est non-fonctionnelle visuellement.

### Problème UX — Import massif sans feedback
300 éléments ajoutés sans indicateur de progression. Refresh pendant l'import → 195/300 éléments, sans notification de l'état partiel. Expérience utilisateur floue sur les opérations longues.

### Gap fonctionnel — Gestion des doublons absente
Aucune détection ni gestion des doublons à l'ajout dans une liste.

### Information — i18n déjà structurée
L'infrastructure i18n existe (`app_fr.arb`, `app_en.arb`, `app_es.arb`, `app_de.arb`) et le sélecteur de langue est présent dans `SettingsPage`. Ce n'est pas à créer from scratch — à vérifier et compléter.

### Fonctionnalité — Insights non exploitable
Module en état vide, non alimenté. À traiter après stabilisation du socle.
